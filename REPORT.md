# Звіт про виконання лабораторної роботи №1: Налаштування Git-workflow для аналізу даних

## 1. Опис проєкту
Проєкт **open-data-ai-analytics** реалізований для демонстрації навичок управління версіями у проєктах з аналізу даних. Основна мета полягала у створенні структурованого репозиторію, роботі з гілками та успішному вирішенні конфліктів.

* **Назва проєкту:** Аналіз пасажирських перевезень АТ "Укрзалізниця".
* **Джерело даних:** [Портал відкритих даних (data.gov.ua)](https://data.gov.ua/)
* **Гіпотези для аналізу:**
    i. Найбільший пасажиропотік у Південно-Західної філії.
    ii. Кількість пасажирів зростає у святкові періоди.
    iii. Дані за 2024 рік мають менше пропусків, ніж за 2023.

## 2. Структура репозиторію
Для проєкту було створено стандартну архітектуру, що забезпечує чіткий поділ даних, коду та звітів:

* `data/` — сирі та оброблені дані.
* `notebooks/` — Jupyter-нотатники для експериментів.
* `src/` — основні Python-скрипти для обробки та аналізу.
* `reports/figures/` — візуалізації та результати аналізу.

## 3. Налаштування Git та Feature Branch Workflow
У роботі використано модель розгалуження **Feature Branching**. Кожен етап розробки виконувався в окремій гілці:
1. `feature/data_load` — реалізація скриптів завантаження.
2. `feature/data_quality_analysis` — перевірка даних на пропуски та аномалії.
3. `feature/data_research` — побудова моделей та основний аналіз.
4. `feature/visualization` — побудова графіків.

## 4. Створення та розв'язання конфлікту злиття
Згідно з завданням, було штучно створено конфлікт у файлі `README.md`.
* **Процес:** Одночасно змінено одну і ту саму секцію у гілках `feature/data_quality_analysis` та `feature/data_research`. При спробі злиття Git видав помилку `CONFLICT (content)`.
* **Вирішення:** Конфлікт був вирішений шляхом ручного редагування маркерів конфлікту та вибору "Accept Both Changes".

## 5. Версіонування та історія
По завершенню всіх етапів розробки було виконано наступні кроки для стабілізації релізу:
* Додано файл `CHANGELOG.md` із описом змін.
* Створено анотований тег версії `v0.1.0` командою: `git tag -a v0.1.0 -m "Перший реліз проєкту"`

## 6. Візуалізація графа Git
Нижче наведено дерево комітів, що демонструє паралельну розробку гілок:

```bash
git log --oneline --graph --decorate --all  
```
## Висновок
Під час виконання роботи було закріплено практичні навички роботи з Git у контексті Data Science проєкту. Налаштовано процес паралельної розробки через Feature Branches, успішно реалізовано стратегію вирішення merge-конфліктів та закладено основу для відтворюваності результатів аналізу.


# Звіт до лабораторної роботи №2

## Тема
Налаштування CI/CD для модульного проєкту

## Мета роботи

Налаштувати автоматичний процес перевірки (CI) та публікації результатів (CD) для модульного проєкту з аналізу даних, а також ознайомитися з роботою self-hosted runner.

## Хід роботи

### 1. Налаштування CI (Continuous Integration)

Було створено workflow `ci.yml` у директорії `.github/workflows`.

Pipeline запускається у таких випадках:
- при `push` у гілку `main`;
- при створенні `pull_request` у `main`;
- вручну через `workflow_dispatch`.

### 2. Запуск модулів

Проєкт складається з таких модулів:

- `data_load` — завантаження даних;
- `data_quality` — перевірка якості даних;
- `data_research` — аналіз даних;
- `visualization` — побудова графіків.

Для паралельного запуску використано **matrix strategy**.

Кожен модуль виконується окремо та проходить такі етапи:
1. Отримання коду (checkout).
2. Налаштування Python середовища.
3. Встановлення залежностей.
4. Запуск відповідного скрипта.
5. Збереження результатів у папку `artifacts/`.
6. Завантаження результатів через GitHub Actions.

### 3. Артефакти (Artifacts)

Результати виконання кожного модуля (логи, файли, звіти) автоматично зберігаються як **artifacts**.

Це дозволяє:
- переглянути результати без запуску локально;
- зберігати історію виконань;
- спростити аналіз помилок.

### 4. Self-hosted runner

Було додано окремий workflow `ci-selfhosted.yml`, який запускає модуль `data_quality` на локальному комп’ютері користувача.

Це дозволяє використовувати:
- локальні ресурси;
- великі датасети;
- власне середовище виконання.

---

## Порівняння runner'ів

### GitHub-hosted runner

**Переваги:**
- не потребує налаштування;
- стабільний та доступний;
- чисте середовище при кожному запуску;
- безпечніший.

**Недоліки:**
- немає доступу до локальних файлів;
- повільніший запуск через встановлення залежностей;
- обмежені ресурси.

---

### Self-hosted runner

**Переваги:**
- доступ до локальних даних;
- швидший запуск (якщо середовище вже налаштоване);
- можливість використовувати великі датасети.

**Недоліки:**
- потрібно налаштовувати вручну;
- ризики безпеки;
- може бути недоступний (offline);
- залежить від стану локального ПК.

---

У ході лабораторної роботи було налаштовано CI/CD процес для модульного проєкту.  
Реалізовано автоматичний запуск перевірок, збереження результатів та запуск задач на self-hosted runner.  
Отримано практичні навички роботи з GitHub Actions та автоматизації процесів розробки.
